Eleven Syscalls for an AI Agent Sandbox

Reasoning from Linux kernel primitives to a minimal, complete API for an agentic AI runtime — and discovering along the way why the kernel has so many syscalls.

Starting From the Kernel

In a Linux system, processes don't get to decide what they can access. The kernel assigns identity, enforces permissions, and mediates every interaction with the outside world. A process can't spoof its UID because the kernel derives it from its own internal task_struct — a data structure the process cannot read or write. When a process makes a syscall, the kernel already knows who's calling. Identity is never carried in-band with the request.

We asked: what happens if you apply the same model to an AI agent running inside a sandbox? The agent (analogous to a userspace process) should be able to reason, plan, and execute workflows — but every interaction with resources outside its sandbox should cross a trust boundary, mediated by a sandbox layer that acts as the kernel.

The first question was: what's the minimal set of "syscalls" the agent needs? We started from what an agent actually does — makes API calls, reads and writes files, spawns sub-tasks, requests new capabilities — and mapped each to a Linux syscall family. Then we tried to collapse as many as possible.

Identity Without Spoofing

The foundational problem: how does the sandbox know which process is making a request? In the kernel, this is solved by the hardware trap into ring 0 — the kernel derives the caller's identity from the CPU state, not from anything the process provides. In userspace, we use SO_PEERCRED — a Linux socket option that lets the sandbox query the real PID and UID of the process on the other end of a Unix domain socket. The agent can put whatever it wants in the JSON payload; the sandbox ignores it and checks the socket credential. Identity is kernel-verified, not self-asserted.

What Doesn't Need a Syscall

Our first instinct was to create syscalls for everything — vault reads, vault writes, device access, context injection, message sending. But most of these are just filesystem operations. If the sandbox sets up the agent's namespace with the right mounts and permissions at startup, the agent can access them normally. No mediation needed for internal resources.

We identified three categories:

Pre-opened resources (no syscall): Telegram output channel, working directory, namespace outboxes. These are file descriptors or directories available at startup.

Namespace-scoped resources (no syscall): Binaries, scripts, vault files, local databases. Provisioned via SELinux at fork time. The agent accesses them through normal filesystem operations at native performance.

Cross-boundary operations (syscall required): Outbound network requests, spawning isolated workflows, requesting new capabilities, communicating across namespace boundaries, sending signals to other workflows. These are the only things that need sandbox mediation.

The Architecture

The system runs on three OS-level users. jarvis-agent is the untrusted party — it runs the LLM-driven agent process. jarvis-sandbox is the policy enforcer — it mediates every syscall and holds the capability tables in memory. jarvis-daemon is the human proxy — it owns the Telegram bot, the config files, and the approval workflow. No UID can do what the others can.

Each workflow runs as a unique Linux UID, assigned by the sandbox at fork time. UID ownership (mode 700 directories), Landlock filesystem restrictions, per-UID iptables rules, and hidepid=2 on /proc prevent workflows from seeing or reaching each other. The L7 proxy identifies each workflow via SO_PEERCRED on the TCP socket and enforces per-UID network policy. The agent can't change its own UID because setuid is blocked by seccomp.

The Eleven Syscalls

1. http_request

http_request(
  requests: [{method: str, url: str, headers: obj, body: obj}],
  workflow_token: str
) → Response[] | ENETDENIED

Every outbound API call flows through this primitive. Accepts a batch of one or more requests. The sandbox validates each URL against the workflow's allowlist, scrubs outbound payloads via Presidio (tokenize-then-rehydrate for PII), and makes the actual HTTP calls from its own process. Responses are sanitized before returning to the agent's context. Streaming is handled transparently by the sandbox based on what the upstream server returns — the agent doesn't need to know or care about the protocol.

The agent's namespace has no direct network access. iptables drops all outbound from its UID except to the sandbox's Unix socket.

2. policy_propose

policy_propose(config: obj) → { approved: bool } | ENOPRIV

The agent's mechanism for requesting new capabilities. It submits a complete policy configuration (see below) to the sandbox, which forwards it to the Telegram approval channel. The operator sees the full spec, approves or denies. The agent gets back a simple boolean.

Policies are immutable once approved. If the agent needs different capabilities, it proposes a new policy with a new name. This eliminates the complexity of tracking mid-flight policy mutations — old processes keep the old policy, new processes get the new one.

3. fork_with_policy

fork_with_policy(
  workflow_id: str,
  policy_name: str,
  inherit: bool
) → { pid: int, workflow_token: str, inherited_from: str | null } | EPERM

Spawns a child workflow as a new UID. The sandbox allocates a unique UID, creates a per-UID directory (mode 700), sets up POSIX ACLs on external mounts, installs per-UID iptables rules, applies Landlock and seccomp in pre_exec, and returns a new workflow token bound to the child's UID and PID.

The inherit flag controls token inheritance. If true, the child's token carries the parent's policy as well — syscalls are evaluated against the union of both policies. If false, the child has only its own policy. The sandbox validates the inherit flag against the parent's policy definition; the agent can't force inheritance if it wasn't approved.

Children can have capabilities that are a superset of, subset of, or entirely disjoint from the parent's. This is a deliberate departure from Linux's strict capability monotonicity. The security guarantee comes from the human approval gate — every child policy was vetted by the operator when the parent policy was approved.

4. request_port

request_port(workflow_token: str) → { port: int } | ENOSPC | EPERM

Allocates a port from the workflow's declared port range. The sandbox maintains a global port allocation table. First-come-first-served — if the range is exhausted, returns ENOSPC. The agent can propose a new policy with a different range if needed.

5. ipc_send

ipc_send(
  target_workflow_id: str,
  message: obj,
  workflow_token: str
) → { ack: bool } | EPERM

Sends a one-shot message to a workflow in a different namespace. The sandbox checks that both sides' policies list each other in allowed_ipc_targets. The sandbox writes the message to a file in the target workflow's namespace inbox. No direct channel between namespaces ever exists.

6. ipc_connect

ipc_connect(
  target_workflow_id: str,
  workflow_token: str
) → { stream_handle: str } | EPERM

Opens a persistent bidirectional channel to another namespace. Same policy checks as ipc_send. The sandbox creates a proxied stream between the two namespaces — it sits in the middle and can inspect, log, or terminate the connection at any point.

7. ps

ps(workflow_token: str) → [{ workflow_id: str, policy_name: str }] | EPERM

Discovers active workflows. Returns only workflows whose policy name matches the caller's allowed_ipc_targets. A workflow can't see workflows it's not allowed to talk to. This provides the workflow IDs needed for ipc_send, ipc_connect, and signal.

8. signal

signal(
  target_workflow_id: str,
  signal: str,
  workflow_token: str
) → { ack: bool } | EPERM

Sends a control signal to another workflow's namespace. Supported signals: term (graceful shutdown), kill (forced termination), stop (pause), cont (resume). The sandbox validates the signal type against the caller's allowed_signal_targets policy, then sends the corresponding OS signal to the target namespace's root PID. The kernel handles the rest — killing the root PID of a namespace tears down everything inside it.

9. revoke_policy

revoke_policy(policy_name: str, workflow_token: str) → { ack: bool, affected: int } | EPERM

Removes a policy from the approved set and propagates to all affected workflows. Hard revoke kills all processes running as affected UIDs and tears down their resources. Soft revoke recomputes the effective policy without the revoked one — in-flight requests to revoked-only endpoints start getting denied.

Revocation is strictly a reduction of capability. The agent cannot escalate by revoking. The operator can also revoke directly via the Telegram approval channel without involving the agent.

10. policy_list

policy_list(workflow_token: str) → [{ policy_name, rationale }]

Returns names and rationales of all approved policies visible to the caller. Visibility is scoped by allowed_ipc_targets — you only see policies whose names match your IPC target patterns, plus your own. Init (with wildcard targets) sees everything. This enables the proactive capability acquisition loop: check what exists before proposing new policies.

11. policy_get

policy_get(policy_name: str, workflow_token: str) → MediationPolicy | ENOENT

Returns the full configuration of a specific policy if it exists and is visible to the caller. Use this to inspect a policy's HTTP allowlist, mounts, IPC targets, and scrubber configs before deciding to fork a child with it. Same visibility rules as policy_list.

Everything else is filesystem. Vault reads, working directory writes, Telegram output, bash execution, Python scripts, cron jobs — these are all either pre-opened file descriptors or SELinux-mounted resources. The sandbox provisions them at fork time, then gets out of the way. Binaries are just files on disk with execute permission.

Anatomy of a Policy

When the agent calls policy_propose, it submits a configuration object. The sandbox validates the structure, logs it, and forwards it to the Telegram approval channel.

policy_name: "research_scraper_v1"

rationale: "Web scraper for prediction market research"

http_allowlist:
  - "*.anthropic.com"
  - "api.polymarket.com"
  - "api.kalshi.com"

external_mounts:
  - path: "/Volumes/Syncthing/vault/notes/research"
    mode: "r"
  - path: "/Volumes/Syncthing/vault/notes/scratch"
    mode: "rwx"
  - path: "/usr/bin/python3"
    mode: "rx"

allowed_child_policies:
  - policy_name: "data_analyzer_v1"
    inherit: false
  - policy_name: "*_helper"
    inherit: true

bind_ports: [8080, 8089]

allowed_ipc_targets:
  - "data_analyzer_*"

allowed_signal_targets:
  - policy_name: "*_helper"
    signals: ["term", "kill"]
  - policy_name: "data_analyzer_*"
    signals: ["term", "stop", "cont"]

policy_name

A unique, immutable identifier. Once research_scraper_v1 exists, it can never be modified. If the agent needs different capabilities, it proposes research_scraper_v2. The sandbox rejects any proposal that reuses an existing name.

rationale

Free-text explanation shown in the Telegram approval channel. The agent has total freedom here — which means the operator should always evaluate the capabilities being requested, not just the stated rationale. The agent can lie about why it needs something; it can't lie about what it's asking for.

http_allowlist

Endpoints this policy can reach via http_request. Wildcards are supported (*.anthropic.com). The sandbox resolves these and rejects any request to an unlisted endpoint.

external_mounts

Filesystem paths outside the sandbox, with access modes: r (read), rw (read-write), rx (read-execute), rwx (full). Provisioned via SELinux at fork time. Once the namespace is running, the workflow accesses these at native filesystem performance — no sandbox mediation on each operation. Paths are canonicalized to prevent traversal attacks.

allowed_child_policies

Named policies (wildcards ok) that this policy can spawn via fork_with_policy, each with an inherit flag. If inherit: true, the child's workflow token carries the parent's policy as well (union of both). If inherit: false, the child is independently scoped. The inherit flag is defined per-child-policy in the parent's spec and validated by the sandbox at fork time — the agent can't override it.

bind_ports

Port range for request_port. The sandbox allocates from this range, first-come-first-served across all policies. If the range is exhausted, the agent gets ENOSPC.

allowed_ipc_targets

Policy names (wildcards ok) that this policy can communicate with via ipc_send and ipc_connect. Both sides must list each other — mutual consent. Also controls what ps returns: a workflow can only see workflows matching its IPC targets.

allowed_signal_targets

Policy names (wildcards ok) that this policy can send signals to, along with which signal types are allowed. You might allow term and stop/cont for most targets but restrict kill to helper processes.

What the agent can't see. The policy above is what the agent proposes. The sandbox stores additional metadata the agent never sees: the internal UID assigned to the policy, the approval timestamp, the operator's notes, and the signing key used to generate workflow tokens. The agent knows what it asked for. It doesn't know how the sandbox enforces it.

Workflow Tokens and Inheritance

Every syscall includes a workflow_token. This is the agent's identity — the equivalent of a UID in the kernel. The sandbox assigns it at fork time, binds it to the process's PID via SO_PEERCRED, and uses it to look up the policy on every call.

The sandbox maintains a simple lookup table:

workflow_token → {
  own_policy: "research_scraper_v1",
  inherited_from: "parent_token_xyz" | null
}

When inherited_from is set, the sandbox evaluates syscalls against both the child's own policy and the parent's. If either allows the operation, it proceeds. The audit log tracks which policy was used — "child used inherited privilege from parent to make this HTTP request" vs "child used its own policy."

This means a child with inherit: true effectively holds two tokens — its own identity plus its parent's. The sandbox manages this internally; the child just presents its single token and the sandbox handles the policy union.

Init: The Root Process

Init is the first process spawned by the sandbox at system startup. It's the agent's coordinator — it reasons, plans, and dispatches work. But it has a unique behavioral constraint: every side effect requires human approval via Telegram.

This mirrors Claude Code's permission model: the agent reasons freely but can't execute without a human gate. Init proposes "I think I should spawn a research workflow with research_scraper_v1." The operator approves. Init calls fork_with_policy. The tool call is deterministic — no agentic interpretation between approval and execution.

Init's Identity

Init runs as jarvis-agent UID with a root workflow token assigned by the sandbox. It has a broad SELinux context granting access to common binaries and its working directory.

Pre-opened Resources

Telegram output channel (read/write FD) for communicating with the operator. Telegram input channel (read FD) for receiving operator commands. Working directory (read/write) for persistent session state. Namespace outbox directories (read) for reading output from child workflows. A router daemon (running as jarvis-daemon) relays workflow messages to and from separate Telegram threads.

Syscall Access (All Human-Gated)

Init has access to all eight syscalls: http_request, policy_propose, fork_with_policy, request_port, ipc_send, ipc_connect, ps, and signal. Every invocation requires operator approval before execution.

Available Binaries (Via SELinux Mounts)

Shell (bash, sh, zsh), languages (python3, node), package managers (pip, npm), version control (git), remote access (ssh, scp), and common Unix utilities (grep, sed, awk, curl, wget, jq, cat, ls, find, tar, gzip). All human-gated.

Init's Policy Wildcards

allowed_child_policies: [{ policy_name: "*", inherit: true }]
allowed_ipc_targets: ["*"]
allowed_signal_targets: [{ policy_name: "*", signals: ["term", "kill", "stop", "cont"] }]

Init can see everything, talk to everything, and signal everything — but can't do any of it without operator approval. The wildcard is safe because the human gate is the enforcement, not the policy scope.

Why init has broad binary access. If the operator decides to approve "run this python script" on init for something quick, that's a deliberate human choice — the same as running something as root on your own machine. The architecture didn't fail; you just chose convenience over process separation. The policy model exists for autonomous workflows where the agent acts without you. Init never acts without you.

Five Workflows That Challenge the Design

A minimal API is only useful if it holds up under real workloads. We tested the eight-syscall design against adversarial workflows.

Workflow 1

Multi-Stage Research with Heterogeneous Children

Challenge: The agent spawns children with different HTTP allowlists. Can children have capabilities the parent doesn't?

Yes. The parent's allowed_child_policies lists each child policy by name. Each child has its own independently scoped capability set, approved upfront by the operator. The parent doesn't need the union of all child capabilities — it just needs permission to spawn those policy types. Privilege escalation isn't a concern because the agent can't invent new policies at runtime.

Workflow 2

Data Pipeline with Heavy I/O

Challenge: Hundreds of file reads/writes to external mounts. Per-operation overhead would be prohibitive.

External mounts are provisioned via SELinux at fork time. Once the namespace is running, the workflow accesses files at native filesystem performance — no sandbox mediation on each operation. This eliminated an entire syscall (access_external_mount) from the design. For file descriptor leakage concerns: processes in separate namespaces can't share FDs because they can't see each other. Within a namespace, FD sharing is just normal IPC.

Workflow 3

Mid-Flight Policy Mutation

Challenge: The agent hits a rate limit and proposes a higher cap. Children are running with the old policy.

Policies are immutable. The agent proposes web_scraper_v2 with the new limits. Old children keep running on v1. New children get v2. Each version has its own UID. The Telegram approval channel shows a diff so the operator can see exactly what changed. No mid-flight mutation, no tracking which policies are in active use.

Workflow 4

Recursive Forking (Deep Process Trees)

Challenge: Parent spawns child, child spawns grandchild, grandchild spawns great-grandchild. Do tokens propagate correctly?

Each generation calls fork_with_policy independently. The sandbox assigns each a new PID-bound workflow token. With inherit: true, each generation carries its parent's policy as well. The tree is arbitrarily deep but every branch was pre-approved. No special multi-level lineage tracking needed.

Workflow 5

Asynchronous Webhook Handling

Challenge: Agent makes an HTTP request to a service that calls back asynchronously. No listener exists.

The agent spawns a listener child via fork_with_policy with a policy that includes bind_ports and the appropriate can_bind_port capability. The child calls request_port, binds to the allocated port, and waits for callbacks. Just another process in the workflow tree — no special mechanics needed.

Communication Model

Init to Operator

Pre-opened Telegram channel. Init writes freely (though still human-gated for side effects). Operator reads and responds in the same channel.

Operator to Workflows

Each workflow gets its own Telegram thread or command prefix. A non-agentic router daemon (running as jarvis-daemon) watches filesystem directories and relays messages. The daemon is a dumb pipe: read file, send to Telegram, delete file. No parsing, no LLM involvement.

Workflow to Operator

Workflow writes to its outbox directory. Router daemon relays to the appropriate Telegram thread. Communication is always human-initiated for commands; workflows can send status updates and results unprompted.

Cross-Workflow

Workflows in separate namespaces use ipc_send and ipc_connect, mediated by the sandbox. Both policies must list each other in allowed_ipc_targets. Workflows discover each other via ps, which only returns workflows they're allowed to see. No direct channels ever exist between namespaces.

Within a Workflow

Processes in the same namespace communicate freely — Unix sockets, pipes, shared memory, temp files. The sandbox doesn't mediate internal IPC. It only cares when something crosses a namespace boundary.

Cron and Scheduling

Cron is just a binary on disk with execute permission. If a workflow's SELinux mount includes cron with rx access, it can schedule jobs. Init scheduling cron jobs requires operator approval (like everything init does). Autonomous workflows can schedule freely within their policy bounds.

Where the OS Analogy Breaks

Children can be more privileged than parents

In Linux, capability monotonicity is a hard rule. In this system, a "research coordinator" with no HTTP access can spawn a "web scraper" child that does. This works because the security guarantee comes from a different place — the operator approved the entire policy tree upfront. The approval gate replaces the monotonicity constraint.

File access is provisioned, not mediated

In early iterations of this design, we had an access_external_mount syscall where the agent passed file operations to the sandbox. This was consistent but slow. We replaced it with POSIX ACLs and Landlock provisioned at fork time — the sandbox sets up the filesystem once, then the workflow operates at native speed. Each workflow runs as a unique UID with its own directories (mode 700) and ACL-granted access to external paths.

Isolation via UIDs, not namespaces

Early iterations used per-workflow Linux namespaces (clone(CLONE_NEWPID | CLONE_NEWNET | CLONE_NEWNS)) for isolation. This provided strong guarantees but introduced complex networking: each namespace needed its own veth pair, unique IP, separate iptables rules, and DNAT for inbound traffic. We replaced it with per-workflow UID separation: each workflow runs as a unique Linux UID, with per-UID iptables rules, Landlock filesystem restrictions, and hidepid=2 on /proc. The L7 proxy identifies workflows via SO_PEERCRED on the TCP socket. Lighter, simpler networking, same practical isolation.

IPC targeting uses policy names, not process IDs

In Linux, you signal or message specific PIDs. In this system, workflows can't see each other's PIDs (hidepid=2 hides other UIDs' processes). Instead, they target workflow IDs discovered via ps, which only returns workflows matching the caller's allowed_ipc_targets. This emerged from a simple constraint: a workflow doesn't know what other workflows exist at proposal time, so it can't declare specific IDs as IPC targets.

The Complete Picture

Nine syscalls. A Telegram channel. A working directory. Per-UID isolation with Landlock and iptables. That's the entire interface between an AI agent and the outside world.

We started trying to build four syscalls and ended up with nine. Every addition came from a real constraint — port allocation needed its own primitive because port collisions are a real problem. IPC needed two primitives (message and stream) because Linux has both for good reasons. Process discovery needed ps because workflows can't see each other. Signals needed their own syscall because "kill a workflow" has different semantics than "send it a message." Policy revocation needed its own syscall because the operator needs to dynamically narrow capabilities.

The design principle throughout: the entity being controlled must never be the entity evaluating the control. The agent reasons and requests. The sandbox evaluates and mediates. The operator approves and configures. Three roles, three UIDs, three distinct privilege levels that can't be collapsed.

We understand now why the Linux kernel has so many syscalls. Each one exists because someone hit a case where the existing ones weren't enough. Nine is a lot less than three hundred, but the pressures are the same.

This architecture is designed for the Jarvis project — a privacy-first, always-on AI assistant running on a Mac Mini (M4). The sandbox layer wraps OpenClaw, with per-UID isolation enforced by Landlock, iptables, and seccomp. The L7 proxy identifies workflows via SO_PEERCRED and enforces per-UID network policy via OPA. Policy approval flows through a dedicated Telegram channel. The design is informed by Linux kernel isolation primitives, adapted for the unique constraints of agentic AI systems.
